Real-Time Multi-Agent Framework for Detecting Cascading Failures in ML Pipelines

A real-time system that monitors the ML pipeline at every stage and provides early alerts for cascading failures.

Project Overview

Modern machine learning pipelines are multi-stage systems where outputs from one stage become inputs to the next. Small issues like data quality problems, model performance drops, or service delays can propagate across stages causing large-scale failures. Traditional monitoring tools often ignore these interactions and operate offline. This project implements a multi-agent framework that monitors each stage in real time, estimates system-level failure risk, and triggers early warnings.

Objectives

  • Monitor Data, Model, and API stages in real time
  • Detect cascading failures before large-scale system impact
  • Visualize propagation risk across the pipeline
  • Provide early alerts and proactive insights
  • Allow scenario injections to test system robustness

Visualization Mapping to Objectives

The dashboard visualizations are designed to clearly show the current system health and propagation of failures:

  • Data Risk Graph: Displays upstream anomalies and drift in real time.
  • Model Risk Graph: Shows amplification of risk from upstream data issues.
  • API Risk Graph: Shows downstream service impact from model degradation.
  • Cascade Risk Graph: Aggregates multi-agent signals to indicate systemic risk.
  • Pipeline Animation: Nodes pulse and arrows glow dynamically, visually demonstrating risk propagation.
  • Risk Gauge: Shows numeric cascade risk in real time with color-coded safety levels.
  • Scenario Panel: Allows injection of drift, model collapse, or API latency, showing how the system reacts.
  • Live Log Panel: Displays agent reasoning and explanations for observed risk.

How Visualizations Relate to Project Goals

Each component on the dashboard corresponds to an objective:

  • Graphs demonstrate the monitoring of each stage.
  • Animations demonstrate propagation effects in real time.
  • Forecasted risk lines show proactive prediction capabilities.
  • Alerts and gauge provide immediate actionable insights.

Scenario Explanation

Scenario injection lets you simulate:

  • Data Drift: Mimics upstream data anomalies.
  • Model Collapse: Simulates model underperformance due to bad data.
  • API Latency: Tests downstream system reliability under stress.

All scenario changes dynamically update the graphs, pipeline animations, and risk gauge, demonstrating the real-time response of the multi-agent system.

Critical Questions & Answers

Q1: What is the main purpose of this project?

A: To detect cascading failures across multi-stage ML pipelines in real time using a multi-agent system.

Q2: What agents are used in the system?

A: Data Agent, Model Agent, API Agent, Propagation Risk Agent, and Alert Agent.

Q3: How does the Data Agent detect anomalies?

A: It monitors statistical deviations from baseline, including Z-score, KL divergence, and scenario injections.

Q4: How is model risk computed?

A: Model risk is derived from amplified data risk and evaluated using historical cascade memory and propagation factors.

Q5: How is API risk computed?

A: API risk is computed based on model risk amplification, latency spikes, and downstream impact simulation.

Q6: What is cascade risk?

A: Aggregate system-level risk combining data, model, and API risks using weighted factors and historical memory.

Q7: What visualization represents cascade risk?

A: The red curve in the line chart and the risk gauge indicate the systemic cascade intensity.

Q8: How are scenario injections applied?

A: Clicking buttons triggers drift, model collapse, or API latency, dynamically changing risk metrics and visualizations.

Q9: What is Z-score used for?

A: To detect statistical anomalies relative to baseline data distribution.

Q10: What is KL divergence used for?

A: To measure distributional difference between current and baseline data for drift detection.

Q11: How does forecasted cascade risk work?

A: Uses historical cascade trends, slope estimation, and anomaly amplification to predict near-future risk.

Q12: What happens if a critical cascade occurs?

A: Alert panel turns red, and speech synthesis announces a critical cascade for immediate action.

Q13: How does auto-recovery work?

A: If drift timers expire, mean and standard deviation gradually return toward baseline values.

Q14: How is dynamic threshold determined?

A: Based on baseline mean plus a factor of baseline standard deviation to adaptively detect high-risk scenarios.

Q15: How is safety margin calculated?

A: Safety margin = dynamic threshold - cascade risk; determines alert levels.

Q16: What are the alert levels?

A: SAFE, WARNING, CRITICAL, and CASCADE FAILURE depending on safety margin.

Q17: How are historical trends stored?

A: Cascade history array stores previous risk values for prediction and anomaly detection.

Q18: How does live log panel help?

A: Displays agent reasoning, scenario influence, and statistical insights for transparency.

Q19: How does pipeline animation indicate risk?

A: Node glow intensity and arrow color correspond to risk magnitudes in real time.

Q20: What type of data is used?

A: Simulated numeric data with baseline Gaussian distributions and optional injected drifts for testing robustness.

Q21: Why does the graph move continuously?

A: Because new metrics are fetched every 2 seconds, and cascading risk updates dynamically to simulate real-time monitoring.

Q22: How do forecasts improve system monitoring?

A: By showing potential future risks, enabling proactive intervention before critical cascades occur.

Q23: What happens when multiple scenarios are injected simultaneously?

A: Risks combine additively but capped at 100; visualizations reflect cumulative system stress.

Q24: How is real-time data fetched?

A: Frontend fetches from FastAPI /metrics endpoint every 2 seconds using fetch API.

Q25: How are anomalies explained mathematically?

A: Using Z-score, mean/std deviation, and KL divergence calculations shown in the live log panel.

Q26: How is model amplification determined?

A: Model risk = data risk × 1.2; API risk = model risk × 1.15; cascade risk is weighted aggregation.

Q27: How does auto-recovery affect risk graphs?

A: Gradually reduces artificially inflated risks toward baseline; green zone returns over time.

Q28: How are scenario effects visualized?

A: Dashboard updates graphs, node pulses, arrows, and logs to show real-time impact of injected scenarios.

Q29: How is forecasted cascade risk plotted?

A: Green dashed line in chart shows predicted risk; dynamically updated alongside actual metrics.

Q30: What uniqueness does this project have?

A: Real-time multi-agent monitoring of cascading failures with explainable reasoning, forecasting, and scenario simulation.

Q31: How are drift types handled?

A: Mean shift, variance explosion, sudden shock, and gradual drift; each modifies the simulated data distribution differently.

Q32: How is the forecast anomaly detected?

A: Using Z-score > 2 compared to historical cascade history; amplifies predicted cascade risk.

Q33: How are frontend animations synchronized with metrics?

A: JavaScript updates CSS properties of nodes and arrows each update cycle to reflect current risk values.

Q34: How does this project demonstrate proactive monitoring?

A: Shows live and predicted risks, triggers alerts before thresholds are exceeded, allowing early interventions.

Q35: What are potential edge cases and how are they handled?

A: Extreme drifts, multiple simultaneous scenarios, and sudden shocks are auto-corrected gradually; risk values capped at 100; alert logic handles safety margin dynamically.
